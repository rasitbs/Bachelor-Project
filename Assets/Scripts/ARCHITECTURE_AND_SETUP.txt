SECTION 1: SYSTEM OVERVIEW
================================================================

The training system uses an event-driven MQTT architecture where:

- MQTTNet library handles all MQTT communication
- Events are published to MQTT broker (backend receives and scores)
- All scoring logic runs on the backend system
- Unity client displays final authoritative score
- No local score tracking exists in the client

KEY CHARACTERISTICS:
- Single persistent MQTT connection for entire application lifetime
- Asynchronous non-blocking operations (does not stall main thread)
- Automatic reconnection with exponential backoff
- Thread-safe message queuing between main thread and background tasks
- Clean separation of concerns via interfaces and services


SECTION 2: ARCHITECTURE DESIGN
================================================================

2A. LAYERED ARCHITECTURE

LAYER 1: MQTT Transport (Core/MQTT/)
  Purpose: Manages connection, publishes messages, receives responses
  Components:
    - MqttApplicationManager: Singleton MQTT client using MQTTNet
    - Handles connection lifecycle and message queue
    - Non-blocking async operations
    - Runs before all other scripts (DefaultExecutionOrder -100)

LAYER 2: Event Publishing (Core/Events/ + Core/MQTT/)
  Purpose: Business logic for event construction and publishing
  Components:
    - EventService: High-level event publishing API
    - MqttEventPublisher: Serializes events to JSON and queues them
    - Provides clean interface for game code (Hazard, HazardTrigger, etc.)
    - Does not know about MQTT details

LAYER 3: Session Management (Core/Session/)
  Purpose: Tracks scenario context stamped on every event
  Components:
    - TrainingSessionManager: MonoBehaviour providing ISessionManager
    - SessionContext: Plain C# class holding session data
    - Generates unique Session ID per scenario run

LAYER 4: Game Logic (Training/)
  Purpose: Publishes events when game rules are triggered
  Components:
    - TrainingScenarioController: Coordinates scenario lifecycle
    - TrainingStateMachine: Tracks gameplay state (NotStarted, Active, Ended)
    - Hazard: Publishes HAZARD_MARKED when player identifies hazard
    - HazardTrigger: Publishes HSE_ALERT_EVENT when player violates safety
    - TrainingUIManager: Displays final score from backend

LAYER 5: Response Handling (Core/Events/)
  Purpose: Receives and processes backend responses
  Components:
    - ScoreResponseHandler: Listens to MQTT and fires score events
    - UI/other systems subscribe to receive final scores


2B. DATA FLOW: SESSION START TO END

Session Initialization:
  1. TrainingSessionManager starts in Training scene
  2. SessionContext generates unique Session ID
  3. EventService finds ISessionManager dependency

Scenario Start:
  1. Player clicks "Start Scenario" button
  2. TrainingScenarioController.StartScenario() called
  3. State machine transitions to Active
  4. EventService.PublishSessionStarted("HMS Grunnleggende")
  5. EventService builds TrainingEvent with session context
  6. MqttEventPublisher serializes to JSON
  7. MqttApplicationManager queues message
  8. Background thread publishes to "events/" topic
  9. Backend receives SESSION_START event

During Gameplay:
  1. Player identifies hazard or violates safety rule
  2. Hazard.Mark() or HazardTrigger.OnTriggerEnter() called
  3. EventService.PublishHazardMarked() or PublishHseAlert()
  4. Events queue and publish asynchronously to MQTT
  5. Backend inserts events into database
  6. Backend maintains running score from events

Scenario End:
  1. Player clicks "End Scenario" button
  2. TrainingScenarioController.EndScenario() called
  3. State machine transitions to Ended
  4. EventService.PublishSessionEnded(duration) publishes SESSION_END
  5. EventService.RequestFinalScore() queues score request
  6. MqttApplicationManager publishes to "request/points" topic
  7. Backend receives SESSION_END and SCORE_REQUEST
  8. Backend calculates final score from all events in this session
  9. Backend publishes response to "response/points" topic
  10. MqttApplicationManager receives on main thread in Update()
  11. ScoreResponseHandler parses JSON and fires OnScoreReceived event
  12. TrainingUIManager displays final score on UI

Session Reset:
  1. Player clicks "Try Again" button
  2. TrainingScenarioController.ResetScenario() called
  3. State machine returns to NotStarted
  4. TrainingSessionManager generates new Session ID
  5. All Hazard and HazardTrigger components reset
  6. Ready for new scenario run


2C. THREADING MODEL

Main Thread:
  - All MonoBehaviour callbacks (Awake, Start, Update, OnTriggerEnter)
  - UI updates
  - Incoming message processing (queued in Update)

Background Thread:
  - MQTT connection establishment
  - Publishing queued messages
  - Network I/O (never blocks main thread)
  - Reconnection logic

Synchronization:
  - ConcurrentQueue for thread-safe message passing
  - Background thread queues received messages
  - Main thread Update() dequeues and processes messages
  - One-way messaging: background puts, main thread gets


SECTION 3: COMPONENT BREAKDOWN
================================================================

3A. MQTT LAYER

Component: MqttApplicationManager (Core/MQTT/MqttApplicationManager.cs)
Purpose: Manages MQTT broker connection and message transport

Responsibilities:
  - Singleton pattern ensures one connection for entire app lifetime
  - Establishes connection on Start() (DefaultExecutionOrder -100)
  - Maintains DontDestroyOnLoad so connection persists across scenes
  - Handles reconnection with exponential backoff on network failure
  - Queues outgoing messages in ConcurrentQueue
  - Receives incoming messages and queues for main thread
  - Provides IsConnected property for status checks

Configuration (Inspector):
  - Broker Host: IP address or hostname of MQTT broker
  - Broker Port: Typically 1883 for MQTT
  - Connection Timeout Ms: Milliseconds to wait for connection
  - Reconnect Delay Ms: Initial delay before reconnect attempt
  - Topic Event Publish: Default "events/" for event publishing
  - Topic Score Request: Default "request/points" for score requests
  - Topic Score Response: Default "response/points" for score responses

Events:
  - OnMessageReceived: Fired on main thread when message received
    Parameters: (string topic, string payload)
  - OnConnectionStatusChanged: Fired when connection established/lost
    Parameters: (bool isConnected)

Methods:
  - Publish(string topic, string payload): Queue message for async publishing
  - SubscribeAsync(string topic): Subscribe to MQTT topic
  - IsConnected: Property indicating if broker is connected

Thread Safety:
  - ConcurrentQueue for publish queue
  - ConcurrentQueue for message reception queue
  - All access is thread-safe
  - Main thread processes received messages in Update()


Component: MqttEventPublisher (Core/MQTT/MqttEventPublisher.cs)
Purpose: Serializes events and publishes via MqttApplicationManager

Responsibilities:
  - Implements IMqttEventPublisher interface
  - Converts TrainingEvent objects to JSON using JsonUtility
  - Publishes JSON payloads to configured topics
  - Null checks and error handling

Configuration (Inspector):
  - Topic Event Publish: Where to send training events (default "events/")
  - Topic Score Request: Where to send score requests (default "request/points")

Methods:
  - PublishEvent(TrainingEvent evt): Serialize and publish training event
  - PublishScoreRequest(ScoreRequestMessage request): Publish score request


3B. EVENT SERVICE LAYER

Component: EventService (Core/Events/EventService.cs)
Purpose: High-level API for publishing training events

Responsibilities:
  - Singleton providing clean event publishing interface
  - DontDestroyOnLoad across scene loads
  - Resolves dependencies: ISessionManager and IMqttEventPublisher
  - Builds structured TrainingEvent objects
  - Timestamps events with UTC datetime
  - Includes scene information in headers

Configuration:
  - No inspector fields required
  - Automatically finds implementations at Start()

Dependency Injection:
  - Searches for ISessionManager implementation (TrainingSessionManager)
  - Searches for IMqttEventPublisher implementation (MqttEventPublisher)
  - Logs warnings if dependencies not found

Methods:
  - PublishSessionStarted(string scenarioName)
    - Publishes SESSION_START event
    - Called at beginning of scenario
  
  - PublishSessionEnded(float duration)
    - Publishes SESSION_END event with duration
    - Called at end of scenario
  
  - PublishHazardMarked(string hazardId, bool correct, int points)
    - Publishes HAZARD_MARKED event
    - Called by Hazard component when player marks hazard
  
  - PublishHseAlert(string triggerId, string description, int penalty)
    - Publishes HSE_ALERT_EVENT
    - Called by HazardTrigger when player enters unsafe zone
  
  - PublishActionInteract(string targetObjectId, string action, string toolUsed, 
                         bool isSuccess, string reason, string consequence)
    - Publishes ACTION_INTERACT event
    - For custom interactive objects
  
  - RequestFinalScore()
    - Queues SCORE_REQUEST to backend
    - Backend responds with final score
    - Should be called when scenario ends

Internal: BuildEvent()
  - Helper method constructing TrainingEvent with headers
  - Timestamps with System.DateTime.UtcNow
  - Includes session context from ISessionManager
  - Includes scene ID from SceneManager


Component: ScoreResponseHandler (Core/Events/ScoreResponseHandler.cs)
Purpose: Receives and dispatches score responses from backend

Responsibilities:
  - Listens to MqttApplicationManager.OnMessageReceived
  - Filters for configured response topic
  - Deserializes JSON to ScoreResponseMessage
  - Fires OnScoreReceived event with session ID and score

Configuration (Inspector):
  - Topic Score Response: Which topic to listen on (default "response/points")

Events:
  - OnScoreReceived: Fired when score received from backend
    Parameters: (string sessionId, int finalScore)
    - sessionId: Identifier of the session being scored
    - finalScore: Authoritative score calculated by backend

Usage Pattern:
  private void OnEnable()
  {
      ScoreResponseHandler.OnScoreReceived += HandleScore;
  }
  
  private void OnDisable()
  {
      ScoreResponseHandler.OnScoreReceived -= HandleScore;
  }
  
  private void HandleScore(string sessionId, int finalScore)
  {
      Debug.Log($"Score received: {finalScore}");
  }


3C. SESSION MANAGEMENT LAYER

Component: TrainingSessionManager (Core/Session/TrainingSessionManager.cs)
Purpose: Manages training session context

Responsibilities:
  - Implements ISessionManager interface
  - Creates SessionContext on Awake()
  - Generates unique Session ID on Start()
  - Provides scenario ID and name
  - Handles session reset for new runs

Configuration (Inspector):
  - Scenario ID: Unique identifier for scenario (e.g., "SCENARIO-001")
  - Scenario Name: Display name (e.g., "HMS Grunnleggende")

Methods:
  - GetSessionId(): Returns unique ID for current session run
  - GetScenarioId(): Returns configured scenario ID
  - GetScenarioName(): Returns configured scenario name
  - ResetSession(): Generates new Session ID and resets context

Properties:
  - ScenarioId: Configured scenario identifier
  - ScenarioName: Configured scenario name


Component: SessionContext (Core/Session/SessionContext.cs)
Purpose: Plain C# class holding session data (not a MonoBehaviour)

Responsibilities:
  - Generates unique Session ID via Guid
  - Records scenario and scene identifiers
  - Provides clean data holder without lifecycle concerns

Properties:
  - SessionId: Unique identifier (Guid format, no hyphens)
  - ScenarioId: Scenario identifier string
  - ScenarioName: Scenario name string
  - SceneId: Active scene build index

Methods:
  - Start(string scenarioId, string scenarioName): Initialize session
  - Reset(): Clear all session data


3D. EVENT DATA MODELS

All defined in Core/Events/EventModel.cs and marked [Serializable] for JsonUtility:

EventHeader:
  - sessionId: string - Unique session identifier
  - timestamp: string - UTC datetime "yyyy-MM-ddTHH:mm:ss.fffZ"
  - sceneId: int - Scene build index
  - eventType: string - Event type identifier

EventPayload:
  - Dynamic fields based on event type
  
  SESSION_START:
    - scenarioName: string
  
  SESSION_END:
    - finalScore: int
    - duration: float
  
  ACTION_INTERACT:
    - targetObjectId: string
    - action: string
    - toolUsed: string
    - isSuccess: bool
  
  HAZARD_MARKED:
    - hazardId: string
    - correct: bool
    - points: int
  
  HSE_ALERT_EVENT:
    - triggerId: string
    - description: string
    - penalty: int

TelemetryData:
  - currentScore: int (always 0 on client; backend calculates)
  - playerPosition: Vector3Data (x, y, z floats)

TrainingEvent:
  - header: EventHeader
  - payload: EventPayload
  - telemetry: TelemetryData

ScoreRequestMessage:
  - sessionId: string
  - type: string = "SCORE_REQUEST"

ScoreResponseMessage:
  - sessionId: string
  - finalScore: int


3E. GAME LOGIC LAYER

Component: TrainingScenarioController (Training/TrainingScenarioController.cs)
Purpose: Coordinates training scenario lifecycle

Responsibilities:
  - Manages scenario state via TrainingStateMachine
  - Triggers event publishing at key lifecycle points
  - Tracks scenario duration
  - Coordinates scenario reset

Configuration (Inspector):
  - State Machine: Reference to TrainingStateMachine component

Dependencies:
  - TrainingStateMachine (auto-found in children)
  - TrainingSessionManager (found via FindFirstObjectByType)
  - EventService (uses static Instance)

Methods:
  - StartScenario()
    - Validates not already started
    - Sets state to Active
    - Records start time
    - Publishes SESSION_START event
    - Called from UI button or game logic
  
  - EndScenario()
    - Validates scenario is active
    - Calculates duration
    - Sets state to Ended
    - Publishes SESSION_END event
    - Calls RequestFinalScore() to request backend calculation
    - Called from UI button or game logic
  
  - ResetScenario()
    - Returns state to NotStarted
    - Resets TrainingSessionManager (generates new Session ID)
    - Resets all Hazard components
    - Resets all HazardTrigger components
    - Called from "Try Again" button or game logic


Component: TrainingStateMachine (Training/TrainingStateMachine.cs)
Purpose: Minimal state machine tracking scenario phase

Responsibilities:
  - Tracks three-state flow: NotStarted -> Active -> Ended
  - Provides IsActive convenience property
  - Purely for local gameplay flow (not event publishing)

States:
  NotStarted: Initial state before scenario starts
  Active: Player is interacting with training content
  Ended: Scenario has concluded

Methods:
  - SetState(TrainingState newState)
    - Only transitions if different from current
    - Logs state changes in editor
  
Properties:
  - CurrentState: TrainingState enum value
  - IsActive: Boolean convenience (true when CurrentState == Active)


Component: Hazard (Training/Hazard.cs)
Purpose: Represents a hazard for the player to identify

Responsibilities:
  - Publishes HAZARD_MARKED event when player marks it
  - Prevents duplicate marking via _marked flag
  - Implements anti-spam cooldown
  - Provides visual feedback (color change on renderer)
  - Implements IHazardComponent for reset capability

Configuration (Inspector):
  - Hazard ID: Unique identifier (e.g., "H-001")
  - Is Correct Hazard: Boolean (true for correct, false for false positive)
  - Points: Integer points value (positive for scoring)
  - Target Renderer: Renderer for visual feedback (auto-found in children)
  - Cooldown Seconds: Time between marks to prevent spam

Behavior:
  - One-shot marking prevented by _marked flag
  - Prevents multiple calls within cooldown window
  - Only fires when state machine is Active
  - Publishes positive points if correct, negative if wrong
  - Changes renderer color: green for correct, red for incorrect

Methods:
  - Mark()
    - Called when player identifies hazard
    - Validates only once per scenario
    - Publishes HAZARD_MARKED event
    - Changes renderer color
  
  - Reset()
    - Called by TrainingScenarioController.ResetScenario()
    - Clears _marked flag
    - Resets color to white


Component: HazardTrigger (Training/HazardTrigger.cs)
Purpose: Detects when player violates safety rule (enters unsafe zone)

Responsibilities:
  - Publishes HSE_ALERT_EVENT when player enters trigger
  - Supports one-time firing or repeated firing
  - Implements anti-spam cooldown
  - Filters by tag to only trigger on specific objects
  - Implements IHazardComponent for reset capability

Configuration (Inspector):
  - Trigger ID: Unique identifier (e.g., "T-001")
  - Description: Reason for penalty (e.g., "Entered live equipment zone")
  - Penalty Points: Integer penalty value
  - Only Once: Boolean (true for one-time, false for repeated)
  - Cooldown Seconds: Time between triggers to prevent spam
  - Required Tag: Tag filter (empty = all objects, "" = specific tag)

Behavior:
  - OnTriggerEnter fires if conditions met
  - Only fires when state machine is Active
  - Optionally fires only once per session (onlyOnce flag)
  - Prevents rapid re-firing via cooldown
  - Publishes negative points (penalty)

Methods:
  - OnTriggerEnter(Collider other)
    - Called by physics engine when collider enters trigger
    - Validates all conditions before firing
    - Publishes HSE_ALERT_EVENT
  
  - Reset()
    - Called by TrainingScenarioController.ResetScenario()
    - Clears _triggered flag
    - Resets cooldown timer


Component: TrainingUIManager (Training/TrainingUIManager.cs)
Purpose: Updates UI based on training events

Responsibilities:
  - Displays final score from backend (via ScoreResponseHandler)
  - Subscribes to ScoreResponseHandler.OnScoreReceived
  - Shows feedback messages and completion status
  - UI is read-only (receives only, does not publish)

Configuration (Inspector):
  - Score Text: TextMeshProUGUI for displaying score
  - State Text: TextMeshProUGUI for displaying state (optional)
  - Feedback Panel: GameObject for feedback visibility toggle (optional)
  - Feedback Text: TextMeshProUGUI for feedback messages (optional)

Behavior:
  - OnEnable subscribes to score response event
  - OnDisable unsubscribes from score response event
  - When score received: updates score text, shows feedback message
  - Feedback automatically hides after 4 seconds

Methods:
  - HandleScoreResponse(string sessionId, int finalScore)
    - Called when backend sends final score
    - Updates score text UI
    - Shows "Scenario Complete" feedback
  
  - ShowFeedback(string message, Color color)
    - Displays temporary feedback message with color
    - Auto-hides after 4 seconds
  
  - HideFeedback()
    - Hides feedback panel


Component: Interfaces and Abstractions

IHazardComponent (Training/IHazardComponent.cs)
  Purpose: Common interface for all hazard-like components
  Properties:
    - HazardId: string - Unique identifier
  Methods:
    - Reset(): void - Reset component state

IXRInteractable (Training/IXRInteractable.cs)
  Purpose: Interface for custom interactive objects
  Properties:
    - ObjectId: string - Unique identifier
  Methods:
    - OnInteractSuccess(string action, string toolUsed)
      - Called on successful interaction
    - OnInteractFailure(string action, string toolUsed, string reason, string consequence)
      - Called on failed interaction

ISessionManager (Core/Events/EventService.cs)
  Purpose: Abstraction for session context provider
  Methods:
    - GetSessionId(): string
    - GetScenarioId(): string
    - GetScenarioName(): string

IMqttEventPublisher (Core/Events/EventService.cs)
  Purpose: Abstraction for MQTT event publishing
  Methods:
    - PublishEvent(TrainingEvent evt): void
    - PublishScoreRequest(ScoreRequestMessage request): void


SECTION 4: INTEGRATION FLOW
================================================================

4A. STARTUP SEQUENCE

1. Application launches
2. Main scene loads with Core infrastructure
3. MqttApplicationManager Awake() (DefaultExecutionOrder -100)
   - Creates singleton instance
   - Sets DontDestroyOnLoad
4. MqttApplicationManager Start()
   - Initiates background connection to MQTT broker
   - Background thread attempts connection with retry
5. MqttEventPublisher Awake()
6. EventService Awake()
   - Creates singleton instance
   - Sets DontDestroyOnLoad
7. EventService Start()
   - Searches for ISessionManager implementation
   - Searches for IMqttEventPublisher implementation
   - Logs if dependencies missing
8. ScoreResponseHandler Awake()
9. ScoreResponseHandler OnEnable()
   - Subscribes to MqttApplicationManager.OnMessageReceived
10. MqttApplicationManager background thread connects
    - Debug logs: "[MQTT] Connected to [host]:[port]"
11. MqttApplicationManager subscribes to response/points topic
12. Training scene loads
13. TrainingSessionManager Awake()
    - Creates SessionContext
14. TrainingSessionManager Start()
    - Initializes session with new Session ID
    - Debug logs: "[SessionManager] Session initialized: [sessionId]"
15. TrainingScenarioController Awake()
    - Finds TrainingStateMachine
16. All other components initialized


4B. EVENT PUBLISHING FLOW

Game code calls:
  EventService.Instance?.PublishHazardMarked(hazardId, isCorrect, points)

Execution sequence:
  1. EventService.PublishHazardMarked() called
  2. Calls internal BuildEvent("HAZARD_MARKED", payload)
  3. BuildEvent creates:
     - EventHeader with session ID, timestamp, scene ID, event type
     - EventPayload with hazard-specific data
     - TelemetryData with score 0 (client doesn't calculate)
  4. Returns TrainingEvent object
  5. Calls MqttEventPublisher.PublishEvent(evt)
  6. MqttEventPublisher calls JsonUtility.ToJson(evt)
  7. MqttEventPublisher calls MqttApplicationManager.Publish(topic, json)
  8. MqttApplicationManager enqueues to _publishQueue
  9. Background thread PublishQueuedEventsAsync() dequeues
  10. Creates MqttApplicationMessage with:
      - Topic: "events/"
      - Payload: JSON bytes
      - QoS: AtLeastOnce
  11. Calls _client.PublishAsync() (non-blocking)
  12. Message sent to MQTT broker
  13. Broker routes to connected backend subscribers
  14. Backend receives and processes event
  15. No acknowledgment needed (fire and forget)


4C. SCORE RESPONSE FLOW

Game code calls:
  EventService.Instance?.RequestFinalScore()

Execution sequence:
  1. EventService.RequestFinalScore() called
  2. Creates ScoreRequestMessage with session ID
  3. Calls MqttEventPublisher.PublishScoreRequest(request)
  4. MqttEventPublisher serializes to JSON
  5. Publishes to "request/points" topic
  6. Background thread sends to broker
  7. Broker routes to connected backend
  8. Backend receives SCORE_REQUEST
  9. Backend calculates final score from all events in session
  10. Backend publishes ScoreResponseMessage to "response/points"
  11. MqttApplicationManager receives message in background thread
  12. Enqueues to _messageQueue
  13. Main thread Update() dequeues message
  14. Fires OnMessageReceived event with topic and payload
  15. ScoreResponseHandler.HandleMessageReceived() called
  16. Filters for "response/points" topic
  17. Deserializes JSON to ScoreResponseMessage
  18. Fires OnScoreReceived event
  19. TrainingUIManager.HandleScoreResponse() called
  20. Updates UI with final score
  21. Shows feedback message


SECTION 5: SCENE SETUP INSTRUCTIONS
================================================================

5A. INSTALL DEPENDENCIES

Step 1: Add MQTTNet Library
  Option A (Recommended - NuGet):
    1. Tools menu -> NuGet Package Manager -> Package Manager Console
    2. Run: Install-Package MQTTnet
    3. Wait for installation to complete
    4. DLL installs to Assets/Plugins/
    5. Restart Unity if needed
  
  Option B (Manual):
    1. Download MQTTnet.dll from https://www.nuget.org/packages/MQTTnet/
    2. Place in: Assets/Plugins/
    3. Set Inspector: CPU = Any, OS = Any
    4. Click Apply


Step 2: Verify Installation
  1. Open Assets/Plugins/
  2. Should see: MQTTnet.dll
  3. Select it, check Inspector: Platform settings = Any


5B. CREATE MAIN SCENE (Infrastructure)

This scene loads first and persists for the entire application.

Steps:
  1. Create new scene, name it "Main"
  2. Add to Build Settings (File -> Build Settings)
  3. Make sure Main scene is FIRST in the list
  4. Create empty GameObject "MqttManager"
  5. Add component: MqttApplicationManager
     - Inspector: Broker Host = your VPS IP
     - Inspector: Broker Port = 1883
     - Inspector: Keep all other defaults
  6. Create empty GameObject "EventPublisher"
  7. Add component: MqttEventPublisher
     - No configuration needed (uses defaults)
  8. Create empty GameObject "CoreServices"
  9. Add component: EventService
  10. Add component: ScoreResponseHandler
  11. Save scene as "Main"


5C. CREATE TRAINING SCENE

This scene contains game logic and training content.

Steps:
  1. Create new scene, name it "Training"
  2. Add to Build Settings (after Main)
  3. Create empty GameObject "TrainingManager"
  4. Add component: TrainingStateMachine
  5. Add component: TrainingScenarioController
     - Drag TrainingStateMachine into State Machine field
  6. Create empty GameObject "SessionManager"
  7. Add component: TrainingSessionManager
     - Inspector: Scenario ID = "SCENARIO-001"
     - Inspector: Scenario Name = "HMS Grunnleggende"
  8. Create Canvas for UI (if not already present)
  9. Add component: TrainingUIManager to Canvas
  10. Create TextMeshProUGUI for score display
      - Parent to Canvas
      - Name: "ScoreDisplay"
      - Drag into TrainingUIManager Score Text field
  11. Save scene as "Training"


5D. CONFIGURE MQTT BROKER CONNECTION

In Main scene:

  1. Select MqttManager GameObject
  2. Expand MqttApplicationManager component
  3. Set Broker Host: Your VPS IP (e.g., "157.245.79.217")
  4. Set Broker Port: 1883
  5. Set Connection Timeout Ms: 5000
  6. Set Reconnect Delay Ms: 5000
  7. Leave topic names as defaults

Testing:
  1. Save scene
  2. Press Play in Unity
  3. Watch Console for messages:
     - "[MQTT] Application manager initialized"
     - "[MQTT] Connected to [host]:[port]"
  4. If connection fails, verify:
     - Broker IP is correct
     - Port 1883 is open on firewall
     - MQTT broker service is running on VPS


5E. CREATE HAZARD OBJECTS

For each hazard in your scene:

  1. Create new GameObject (e.g., "Hazard_Panel_01")
  2. Add Collider component (Box, Sphere, or Capsule)
     - Set Is Trigger: false
     - Size/position to surround hazard visually
  3. Add Hazard component
     - Hazard ID: Unique identifier (e.g., "H-001")
     - Is Correct Hazard: true for correct, false for false positive
     - Points: Positive integer (e.g., 10)
     - Target Renderer: Auto-found from children
     - Cooldown Seconds: 1.0 (prevents spam)
  4. Add visual representation (mesh, sprite, etc.)
  5. Wire up interaction:
     - Option A (XR): Add XRGrabInteractable, On Select Entered -> Hazard.Mark()
     - Option B (Button): Create button UI, On Click -> Hazard.Mark()


5F. CREATE TRIGGER ZONES

For each safety violation zone:

  1. Create new GameObject (e.g., "Zone_HighVoltage")
  2. Add Collider component (Box or Sphere)
     - Set Is Trigger: true
     - Size/position to cover zone
     - Add Rigidbody with Is Kinematic: true
  3. Add HazardTrigger component
     - Trigger ID: Unique identifier (e.g., "T-001")
     - Description: Reason for penalty (e.g., "High voltage equipment")
     - Penalty Points: Positive integer (e.g., 15)
     - Only Once: true for one-time, false for repeating
     - Cooldown Seconds: 1.0 (prevents spam on physics glitches)
     - Required Tag: Player tag (if using tag filtering)
  4. Make visual (optional): semi-transparent red cube


5G. WIRE UP UI BUTTONS

For Start button:
  1. Create Button UI element
  2. Select button GameObject
  3. In Button component, On Click section:
     - Click "+"
     - Drag TrainingManager GameObject

7A. CREATING CUSTOM INTERACTIVE OBJECT

File: Assets/Scripts/Training/BreechBlock.cs

using UnityEngine;

public class BreechBlock : MonoBehaviour, IXRInteractable
{
    public string ObjectId => "breech_block_01";
    
    private bool _isOpen = false;

    public void OnInteractSuccess(string action, string toolUsed = "none")
    {
        _isOpen = true;
        EventService.Instance?.PublishActionInteract(
            ObjectId, action, toolUsed, isSuccess: true
        );
        Debug.Log("Breech block opened successfully");
    }

    public void OnInteractFailure(string action, string toolUsed, 
                                  string reason, string consequence = "")
    {
        EventService.Instance?.PublishActionInteract(
            ObjectId, action, toolUsed, isSuccess: false
        );
        Debug.Log($"Failed: {reason}");
    }

    public void AttemptOpen(string toolUsed)
    {
        if (toolUsed == "insulated_screwdriver")
            OnInteractSuccess("open_breech", toolUsed);
        else
            OnInteractFailure("open_breech", toolUsed, 
                "missing_insulated_screwdriver");
    }
}

SECTION 8: TROUBLESHOOTING
================================================================

PROBLEM: "[MQTT] Application manager initialized" not in console

SOLUTION:
  1. Verify MqttApplicationManager component added to scene
  2. Verify it's in Main scene or first loaded scene
  3. Check DefaultExecutionOrder is -100
  4. Verify MQTTnet.dll is in Assets/Plugins/
  5. Verify script compiles without errors
  6. Force reimport: Assets -> Reimport All


PROBLEM: "Cannot connect to broker"

SOLUTION:
  1. Verify broker IP and port are correct
  2. Test from command line: telnet broker_ip 1883
  3. Verify firewall allows outbound port 1883
  4. Verify MQTT broker service is running on VPS
  5. Check VPS logs for connection attempts
  6. Verify network connectivity to VPS


PROBLEM: Events not publishing

SOLUTION:
  1. Check Console for "[MQTT] Connected" message
  2. Verify EventService.Instance is not null
  3. Verify MqttEventPublisher is in scene
  4. Check MQTT is connected first
  5. Verify game state is Active (not NotStarted or Ended)
  6. Increase log verbosity to see JSON payload


PROBLEM: Score never displays

SOLUTION:
  1. Verify EndScenario() was called
  2. Check Console for "Published SESSION_END"
  3. Verify backend received events and is connected to broker
  4. Check backend is publishing to "response/points" topic
  5. Verify ScoreResponseHandler is in scene
  6. Monitor "response/points" topic on broker for messages
  7. Check JSON format matches ScoreResponseMessage


PROBLEM: Hazard fires multiple times on single click

SOLUTION:
  1. Increase cooldownSeconds to 2.0 or higher
  2. Verify _marked flag is being set
  3. Check collider bounds not overlapping with hand/tool
  4. Verify XR input only fires once per interaction
  5. Disable collider after Mark() if needed


PROBLEM: Trigger zone fires continuously

SOLUTION:
  1. Set HazardTrigger.Only Once = true
  2. Increase cooldownSeconds to 2.0 or higher
  3. Verify collider Is Trigger = true
  4. Verify Rigidbody Is Kinematic = true
  5. Check only one trigger per zone
  6. Add Reset() call when changing scenarios


PROBLEM: Cannot find ISessionManager

SOLUTION:
  1. Verify TrainingSessionManager in Training scene
  2. Check EventService.Start() finds it
  3. Watch Console for warning message
  4. Ensure TrainingSessionManager starts before EventService
  5. Consider setting component execution order


PROBLEM: MQTTnet compilation errors

SOLUTION:
  1. Delete Assets/Plugins/MQTTnet* files
  2. Delete Library folder
  3. Reimport from NuGet: Install-Package MQTTnet
  4. Wait for compilation
  5. Verify MQTTnet.dll in Assets/Plugins/
  6. Restart Unity
